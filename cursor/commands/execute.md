# Execute — Exe Agent

**目的**: 按计划逐步实现功能，始终在 Blueprint 文档的约束下编码，而不只是盯着 Plan 执行。

## 前置条件
- `./plans/[feature-name].md` — 已经过资深工程师 Review 的开发计划

---

## 第 0 步：Mission Briefing（必须首先执行，不可跳过）

在写任何代码之前，读取以下文档建立完整上下文：

```
📋 Mission Briefing 加载中...

1️⃣ ./plans/[feature-name].md
   提取：当前步骤、待完成任务、成功标准

2️⃣ .blueprint/Tech_Spec.md
   提取：
   - 技术栈与版本约束
   - src 目录结构（严格遵守，不自行创建新层级）
   - 数据模型（字段名、类型、关系）
   - API 设计（端点、请求/响应结构）
   - 关键实现细节（状态管理、错误处理、安全要求）

3️⃣ .blueprint/UI_Spec.md + .blueprint/mockup/
   提取：
   - 当前步骤涉及页面的布局结构（以 mockup 为视觉基准）
   - 组件 props 形态与交互状态
   - Design tokens（颜色、间距、字体）

4️⃣ .blueprint/Test_Spec.md
   提取：当前步骤对应的 P0 测试用例 — 这是编码的验收边界

✅ Mission Briefing 完成

当前任务：步骤 [N] — [步骤名称]
验收边界：[对应 Test Spec 用例 ID]
关键约束：[来自 Tech Spec 的关键限制]
```

**任何与 Tech Spec 或 UI Spec 的偏差，必须先暂停讨论，不得自行决定。**

---

## 实施要求

### 代码质量标准
- **优雅**：代码干净、可读、易于理解
- **最小化**：只构建计划里有的内容，不过度工程化
- **模块化**：严格遵守 Tech Spec 定义的 src 目录结构
- **可测试**：编写可以直接对照 Test Spec 验证的代码
- **文档化**：注释解释"为什么"，尤其是复杂逻辑

### 对齐检查点（每个步骤完成前）
在标记步骤为 ✅ 完成前，确认：
- [ ] 数据字段与 Tech Spec 数据模型一致（无自行添加的字段）
- [ ] UI 实现与 mockup 视觉一致（布局、颜色、间距）
- [ ] API 调用与 Tech Spec 接口设计一致（端点、参数、错误处理）
- [ ] 对应 Test Spec 的 P0 用例可以通过

---

## 实施流程

每个步骤：

**开始前**
- 将步骤状态更新为 🟡 进行中
- 重读 Mission Briefing 中与本步骤相关的部分
- 确认要修改/创建的文件列表

**实施期间**
- 遵循 Tech Spec 定义的现有模式
- 在进行时勾选子任务
- 发现与文档不一致之处：**立即暂停并指出，不要自行假设**

**完成后**
- 执行对齐检查点的 4 项确认
- 将步骤状态更新为 ✅ 完成
- 更新总体进度百分比
- 简述完成内容和任何需要后续关注的点

---

## 进度更新格式

```markdown
**总体进度**: `33%`

### ✅ 步骤 1: [步骤名称]
[简述完成内容，及与 Tech Spec/UI Spec 的对齐确认]

### 🟡 步骤 2: [步骤名称]
[当前正在做什么]
- [x] 子任务 1
- [ ] 子任务 2

### ⬜ 步骤 3: [步骤名称]
[待办]
```

---

## 遇到问题时

如果发现：
- 计划中的实现方式与 Tech Spec 有冲突
- UI 需要某个 Tech Spec 没有设计的 API
- 某个需求在 PRD/Tech Spec 中描述模糊

**立即暂停，描述冲突点，等待确认后再继续。** 不要绕过文档自行决定，这是漂移的根源。
