- `.blueprint/UI_Spec.md`

## 维度二：代码质量审查

### 日志记录
- 没有 `console.log` 语句，使用适当的日志记录器
- 日志级别适当（info / warn / error / debug）

### 错误处理
- 异步操作使用 try-catch
- 错误处理得当且信息丰富，格式符合 Tech Spec 定义
- 中心化错误处理

### TypeScript
- 没有 `any` 类型，使用 Tech Spec 中定义的接口/类型
- 类型定义清晰且准确
- 泛型使用得当

### 生产就绪性
- 没有调试语句，没有注释掉的代码
- 没有硬编码的值，使用配置

### React / Hooks
- 副作用有清理，依赖项正确
- 没有无限重渲染的风险

### 性能
- 没有不必要的重渲染
- 昂贵的计算已优化
- 避免不必要的 API 调用

### 安全
- 已检查授权，输入已验证
- 使用 RLS 或适当的授权
- 敏感数据得到保护（符合 Tech Spec 安全要求）

### 架构
- 遵循 Tech Spec 定义的 src 目录结构
- 关注点分离清晰
- 可维护且可扩展

## 问题输出格式

```markdown
#### [严重程度] [文件:行] — [问题描述]
- **类型**: 漂移 / 代码质量
- **问题**: [详细描述]
- **规格依据**: [Tech Spec §X / UI Spec §X / PRD FX]（漂移问题必填）
- **建议**: [建议的修复方案]
- **影响**: [潜在影响]
```

**严重程度级别**：
- 🔴 **严重**：必须在合并前修复（安全漏洞、与规格直接冲突、破坏性 bug）
- 🟡 **中等**：应该修复（性能、可维护性、轻微漂移）
- 🔵 **轻微**：可以考虑（代码风格、小优化）
- 💡 **建议**：可选改进（最佳实践）
